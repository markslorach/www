Most personal sites fail quietly. They still load, the links mostly work, and the homepage says roughly the right things, but the whole place starts to feel like it was last touched in a different version of your life.

I wanted this site to feel smaller, calmer, and easier to update. That meant treating the writing page, project list, and tiny details like the music widget as part of the same system instead of a pile of unrelated sections. It also meant resisting the urge to rebuild everything at once.

## A Small Site Is Still a Product

The useful shift was to stop thinking of the site as a portfolio and start thinking of it as a maintained product. A portfolio tries to impress a stranger in one pass. A product has to keep working after the first pass, including on the days when I only have twenty minutes to publish a note or fix a broken link.

That changed the questions I asked. Instead of asking whether the page looked finished, I asked whether the page made the next edit obvious. A post should be easy to write in `content/writing`, a project should have a clear thumbnail, and the layout should not collapse when a title runs long.

> A personal site does not need to be big. It needs to feel looked after.

That sentence became the test for most decisions. If a feature made the site feel busier without making it easier to maintain, it usually did not survive.

## The Maintenance Problem

The previous version had the usual personal-site problems:

- The homepage was doing too many jobs at once.
- The writing system existed, but publishing still felt like a special event.
- Project summaries were inconsistent, so the list was harder to scan than it needed to be.
- Styling decisions lived in too many places, which made small changes feel risky.

None of those problems are dramatic. They are exactly the sort of friction that makes you postpone updates. When a site is supposed to represent current work, that is enough to matter.

### Make the Default Path Boring

The best improvement was making the happy path intentionally plain. A new article should only need frontmatter, a few headings, and normal Markdown. The styling can do the rest.

For example, the publishing logic should be easy to read at a glance:

```typescript
import { allWritings } from "content-collections";

export function getPublishedArticles() {
  return allWritings
    .filter((article) => article.published)
    .sort(
      (a, b) => new Date(b.date).getTime() - new Date(a.date).getTime(),
    );
}
```

There is nothing clever in that code, which is the point. A personal site has enough surface area without hiding the simple parts behind unnecessary abstractions.

## What Stayed

Some pieces were already working. The sidebar still gives the site a clear owner, the project list still leads with real work, and the dark theme still suits the tone. I kept those parts and spent most of the effort on spacing, hierarchy, and defaults.

The biggest change was more editorial than technical: every component now has to earn its space. **Strong text** should make a sentence easier to scan, not louder. *Emphasis* should feel like a change in tone, not decoration. Links should be obvious enough to find, but quiet enough to read past.

## The Publishing Checklist

Before I publish a new post, I run through a small checklist:

1. Write the rough version in plain Markdown.
2. Check that headings describe the shape of the argument.
3. Add links only where they help the reader leave with more context.
4. Make sure the final paragraph actually lands the point.

That process is deliberately lightweight. If it grows into a production ceremony, I will stop using it.

---

## The Result

The result is not a bigger site. It is a site with fewer loose ends. The article template handles headings, paragraphs, links, quotes, lists, rules, images, captions, inline code, and highlighted code blocks without each post needing custom layout work.

That is the real win. When the system is boring in the right places, the writing can be more specific. I can publish a short note about [a project I am building](https://markslorach.com), document a decision, or leave myself a clear trail for the next round of changes.

The site feels maintained because the maintenance path is now part of the design.
